# ARCH.md 规范

`ARCH.md` 是项目的技术宪法，必须涵盖：

- 整体架构设计与分层模式
- 目录结构说明
- 开发语言与核心依赖库
- 错误处理模式
- **禁止项**：明确不允许出现的写法或依赖

---

## 两种生成模式

### 模式一：源码已存在

用户提供了本地源码路径，执行 `project-analysis` 技能进行扫描分析，从代码中提炼以上各项内容。
禁止项通过观察代码自动推断：识别出作者从未采用的写法，列为禁止项。

### 模式二：空目录（项目尚未开始）

用户提供的目录为空，或项目还未有任何代码。此时通过与用户对话来构建 ARCH.md：

1. **确认意图**：向用户说明目录为空，询问是否基于需求描述来规划技术方案：
   > "目录当前为空，我可以根据你描述的需求帮你规划技术架构，生成初始 ARCH.md，是否继续？"

2. **直接创建 ARCH.md 模板**，将所有待填项以提示语的形式写入文件，交由用户手动补充：

```markdown
# ARCH.md（待用户补充）

## 开发语言 / 技术栈
> 请在此处写明项目使用的开发语言和技术栈，例如：Go 1.21 + Gin + Redis

## 整体架构设计与分层模式
> 请在此处描述项目的整体架构，例如：MVC 三层架构 / Clean Architecture / 微服务

## 目录结构说明
> 请在此处写明预期的目录结构及各目录职责，例如：
> - internal/handler：HTTP 入口层
> - internal/service：业务逻辑层
> - internal/repo：数据访问层

## 核心依赖库
> 请在此处列出主要依赖库及其用途，例如：
> - Gin：HTTP 框架
> - GORM：ORM
> - Zap：日志

## 错误处理模式
> 请在此处说明项目的错误处理约定，例如：统一返回 {code, msg, data} 结构

## 禁止项
> 请在此处列出明确禁止的写法或依赖（可留空）
```

3. **创建完成后告知用户**：
   > "我已在 `projects/{project_name}/ARCH.md` 创建了技术档案模板，请补充完整后把文件拖入对话框，或直接告诉我，我再继续处理需求。"

---

## ARCH.md 末尾必须包含的用户补充模块

生成 ARCH.md 时，末尾必须保留以下模块（内容留空，由用户自行填写）：

```markdown
## ⚠️ 用户补充说明（最高优先级，必须严格遵守）

> 本节由用户手动维护，内容凌驾于所有自动分析结论之上。
> 生成任何 SKILL.md 前，必须完整阅读本节，违反任意一条均视为输出无效。

### 禁止事项
<!-- 例：禁止引入任何 Python 脚本 -->
<!-- 例：禁止使用 ORM，所有数据库操作必须使用原生 SQL -->

### 注意事项
<!-- 例：所有对外接口必须做鉴权 -->
<!-- 例：日志必须使用项目统一的 logger，禁止使用 fmt.Println -->

### 框架说明
<!-- 例：HTTP 框架统一使用 Gin，禁止引入其他 HTTP 库 -->

### 业务说明
<!-- 例：用户 ID 全局唯一，由网关统一生成，服务内不得自行生成 -->

### 其他补充
<!-- 任何不属于以上分类但需要 AI 知晓的信息 -->
```

**生成 `SKILL.md` 前，必须完整读取此模块。此模块任意一条的优先级高于 ARCH.md 其他所有内容。**
