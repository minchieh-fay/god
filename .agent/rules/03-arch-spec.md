# ARCH.md 规范

`ARCH.md` 是项目的技术宪法，必须涵盖：

- 整体架构设计与分层模式
- 目录结构说明
- 开发语言与核心依赖库
- 错误处理模式
- **禁止项**：明确不允许出现的写法或依赖

---

## 两种生成模式

### 模式一：源码已存在

用户提供了本地源码路径，执行 `project-analysis` 技能进行扫描分析，从代码中提炼以上各项内容。
禁止项通过观察代码自动推断：识别出作者从未采用的写法，列为禁止项。

### 模式二：空目录（项目尚未开始）

项目目录下没有源码。此时按以下优先级处理：

#### 优先级 1：需求文件中包含源码路径

若需求文件中明确指出了源码在本地的路径（如："源码在 ~/code/myproject"），直接执行 `project-analysis` 技能分析该路径，自动生成 ARCH.md，跳过后续步骤。

#### 优先级 2：需求文件中包含技术信息

若需求文件中提到了技术栈、框架、语言等信息（如："用 bun + TypeScript 开发"、"基于 Gin 框架"），必须从需求文件中**摘取这些已知信息预填到 ARCH.md**，未提及的字段保留提示语供用户补充：

```markdown
# ARCH.md

## 开发语言 / 技术栈
{从需求文件中摘取，如：Bun + TypeScript}

## 整体架构设计与分层模式
> 请在此处描述项目的整体架构，例如：MVC 三层架构 / Clean Architecture / 微服务

## 目录结构说明
> 请在此处写明预期的目录结构及各目录职责

## 核心依赖库
{从需求文件中摘取已提及的依赖，未提及的留空}
> 其余依赖请在此处补充

## 错误处理模式
> 请在此处说明项目的错误处理约定

## 禁止项
{从需求文件中摘取已提及的禁止项，无则留空}
```

#### 优先级 3：需求文件中无任何技术信息

所有字段均写入提示语，交由用户手动填写。

---

生成完成后，**必须提醒用户确认**：
> "我已根据需求文件生成了 `projects/{project_name}/ARCH.md`，请确认内容是否准确，补充缺失项后告诉我，我再继续生成施工图。"

---

## ARCH.md 末尾必须包含的用户补充模块

生成 ARCH.md 时，末尾必须保留以下模块（内容留空，由用户自行填写）：

```markdown
## ⚠️ 用户补充说明（最高优先级，必须严格遵守）

> 本节由用户手动维护，内容凌驾于所有自动分析结论之上。
> 生成任何 SKILL.md 前，必须完整阅读本节，违反任意一条均视为输出无效。

### 禁止事项
<!-- 例：禁止引入任何 Python 脚本 -->
<!-- 例：禁止使用 ORM，所有数据库操作必须使用原生 SQL -->

### 注意事项
<!-- 例：所有对外接口必须做鉴权 -->
<!-- 例：日志必须使用项目统一的 logger，禁止使用 fmt.Println -->

### 框架说明
<!-- 例：HTTP 框架统一使用 Gin，禁止引入其他 HTTP 库 -->

### 业务说明
<!-- 例：用户 ID 全局唯一，由网关统一生成，服务内不得自行生成 -->

### 其他补充
<!-- 任何不属于以上分类但需要 AI 知晓的信息 -->
```

**生成 `SKILL.md` 前，必须完整读取此模块。此模块任意一条的优先级高于 ARCH.md 其他所有内容。**
