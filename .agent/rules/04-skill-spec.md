# SKILL.md 规范

产出的 `SKILL.md` 必须满足以下所有要求：

## 文件格式要求

必须以 YAML frontmatter 开头，格式如下：

```yaml
---
name: {需求名，小写加连字符，如：add-mail}
description: {一句话描述该技能的用途和触发场景，供 AI 判断何时加载}
---
```

`description` 的写法要求：
- 说明**做什么**以及**在什么情况下使用**
- ✅ `在 mediasvr 项目中添加邮件发送功能。当需要实现邮件通知时使用。`
- ❌ `add mail feature`

---

## 内容结构要求
必须包含以下三个模块：
- `[Context]`：说明当前需求的背景、涉及的文件和模块
- `[Blueprint]`：方案设计，说明做什么、为什么这样做
- `[Implementation Steps]`：逐步实施细节，精确到每个文件的每处修改

## 路径要求
所有文件路径必须使用相对于**目标工程根目录**的相对路径，禁止使用绝对路径。

## 粒度要求
- 单次涉及代码修改量不超过 50 行
- 超过 50 行必须拆分为多个独立任务，每个任务单独生成一个 SKILL.md

## 质量要求
描述精确到：即使最弱的 AI 模型也能准确实现，不产生歧义。

## Implementation Steps 写法规范

`[Implementation Steps]` 必须遵循以下要求：

**1. 按时序排列**
步骤必须严格按照执行顺序排列，每一步完成后才能进行下一步。
禁止跳跃式描述，禁止把"最后"的操作提前出现。

**2. 函数名、变量名必须准确**
所有提到的函数名、方法名、变量名、常量名，必须与目标工程的实际命名一致。
不得使用 `doSomething()`、`handleXxx()` 等模糊占位名称。

**3. 适当加入伪代码**
对于逻辑复杂、容易产生歧义的步骤，必须附上伪代码辅助说明，格式如下：

```
// 伪代码示例
if 用户未登录 {
    返回 401
}
调用 OrderService.createOrder(userId, items)
将结果写入 response.data
```

**4. 每步说明操作对象**
每个步骤需明确指出：在哪个文件、哪个函数、哪个位置进行修改，例如：
- ✅ `在 internal/handler/user.go 的 Login() 函数末尾，追加以下逻辑`
- ❌ `在登录处理函数里加上这段代码`

## 约束检查
生成前必须对照 `ARCH.md` 的「用户补充说明」模块逐条核查，确保无任何违反。
